# Workflows Notes

Disclaimers: 

- The voice agent in* `src/components/`*and some of the scaffolding was generated by my coding agent harness. The supervisor pipeline and Workflow SDK wiring is hand-written. That's where the "core parts" of this take-home actually live. I put those changes in a separate branch/PR to draw the line more clearly.
- I have been experimenting with a voice agent for a while, so this is not a net-new project. Using workflows as the supervisor layer is the focus of this experiment. 

The high-level idea has been for the voice agent to act like a sophisticated "UI" for a number of agents. One of the key challenges for that kind of generic "assistant" is the inherent complexity of handling the conversation flow, the decision logic, and the execution flow all at once. So I have been thinking about separating those concerns: keeping the conversation itself in one agent and handing everything else to a "supervisor" agent.

I decided to test Workflows as the mechanism for that supervisor to coordinate actions with the conversation agent. 
*Sidenote: I'm generally leaning towards an event-based architecture for my use case, aiming to avoid having state baked into the workflow itself. One of the key characteristics of a "supervisor" is that, theoretically, a large number of events, threads and actions may be happening in parallel, so orchestration matters as much as durability (so perhaps closer to the Inngest model).*

The rough outline of the architecture I landed on for the prototype is a Voice loop via OpenAI Realtime (WebRTC) and a Workflow SDK workflow as the "supervisor". The supervisor has two capabilities wired today: `web_search` (Tavily) and `send_message` (one-way Telegram, after a 60s `sleep` for the durability demo).

I came away thinking the directive model is a clean idea that ends up doing more work than the SDK enforces. You can write the wrong shape and only find out at runtime. The composition with OpenAI Realtime worked because the workflow didn't need to know about voice, which I think is a real point in favor of the directive approach. What follows are notes on things I learned, where I tripped, and what I'd still want to ask the team.

## Where to look

- `src/supervisor/process-turn.ts`: `rephrase` and `decideAction` (steps), `processAction` (workflow with `sleep`), `recordEvent` (step wrapping side effects so they don't duplicate on replay).
- `src/app/api/intent/classify/route.ts`: synchronous `rephrase` → fire-and-forget `start(processAction, [...])`.
- `next.config.ts` / `withWorkflow()` wrap. `tsconfig.json` / `workflow` plugin enabled.

Three channels, three jobs:

- **HTTP response body** -> voice-ack. Realtime model reads `ack` from tool-call result. Synchronous, ~500ms budget.
- **Workflow runtime** -> durability. `start()` fire-and-forget. Runs as long as needed, survives lambda churn.
- **SSE / poll** -> dashboard observability. Not consumed by voice.

Conflating any two collapses the model. My first version conflated SSE with the voice-ack channel; the symptom only surfaced once I added `sleep`.

## Findings

1. The voice ack travels in the HTTP response body, not the SSE stream. Realtime tool calls are synchronous, so awaiting the full workflow delays the ack by however long `sleep` runs. Fixed by splitting rephrase out of the workflow and firing the rest as fire-and-forget.
2. There's no spawn primitive. The obvious approach of "fire a child workflow and let the parent return immediately." Workflow SDK doesn't have that. `void someAsync()` inside a workflow works in `pnpm dev` because Node keeps the request handler alive, but breaks in serverless: the lambda terminates when the parent workflow returns, the work disappears. The compiler doesn't warn you. Once I accepted that workflows are linear (one path through the function body, suspendable at step boundaries) and that the event stream is the parallel channel the UI listens on for progress, the absence of spawn stopped feeling like a missing feature. Hooks fill the wait-on-external-input gap (confirmation, webhooks).
3. Steps can't yield. `sleep` lives at workflow scope. Steps are leaves, workflows are the tree. Statically detectable, since the directive is just a string literal in the function body.
4. Workflow invocation has a few prerequisites of its own. For `sleep` to actually mean anything you need `withWorkflow()` in `next.config`, invocation via `start()`, and serializable args. The error `sleep() can only be called inside a workflow function` has four possible root causes and doesn't tell you which one. Would have saved me real time if it did (or if I had read the docs more carefully).
5. Local dev doesn't give you real durability. I tried three different ways of killing the session: ending the WebRTC connection (workflow keeps running because the voice loop and workflow are decoupled), closing the browser tab (workflow keeps running because Node holds the request handler open), and Ctrl-C on the dev server (workflow dies). Only the third tells you anything real about durability: you need a Vercel preview deploy to actually exercise it.
6. Cleanup pass also found: workflow-scope side effects duplicate on replay. Wrap event emits in `"use step"` (`recordEvent` in this repo). SDK doesn't warn about this.

If I had to name the pattern: the SDK lets you write the wrong shape and just tells you at runtime, often only in dev. The directive itself is a string literal in the function body, so a lot of these mistakes look like they could be caught statically.

## Comparison: Workflow SDK vs Temporal vs Inngest

- Three bets on where the safety contract lives: Temporal = replay-based determinism + mature versioning, Inngest = event-driven step memoization + structural cancellation, Workflow SDK = compiler-enforced directive boundaries.
- Inngest is structurally closer to Workflow SDK than Temporal is: both TypeScript-first, both serverless-friendly, both step-memoized. The real difference is event-driven (functions react to events) vs workflow-first (call functions directly).
- Versioning is the hardest unanswered question I have: `sleep("30d")` while you redeploy the workflow code is the case I'd want to interrogate with the team.
- "No lock-in" is true in principle, but the differentiated experience is Vercel-only. Not a criticism — just what you should expect.

## DX suggestions (ranked by leverage)

1. Discriminating error message for `sleep() can only be called inside a workflow function` (four root causes, runtime knows which).
2. Static check for `sleep`/`createHook`/`createWebhook` called from `"use step"` functions.
3. Doc page: AI agent recipe (Realtime tool-call composition).
4. Replay-safety lint for workflow-scope side effects.

## Known issues in this code

- Event log dashboard is flaky: in-process singleton doesn't reliably share state between request handler and workflow runtime contexts. `globalThis` workaround in place; correct fix is shared store.
- No real durability in local dev: close `pnpm dev` mid-sleep and you lose the workflow. Demo durability on Vercel preview.
- Telegram is one-way send only; inbound webhook + `createHook` resume scoped out for time.
- Cancel is cosmetic: `cancel_pending_action` Realtime tool acks the model but doesn't reach the workflow.

## What I'd build next

1. Bidirectional Telegram with `createHook` confirm-via-reply.
2. Real cancel via `actionId → runId` map + `getRun().cancel()`.
3. Barge-in retry as workflow-managed delivery.
4. Multi-turn disambiguation via `createHook`.
5. Per-capability Realtime tools instead of one generic `classify_and_route`.
